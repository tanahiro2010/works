# SQLインジェクション PoC サンドボックス

このフォルダは講義スライド「SQLインジェクション ――攻撃の仕組みから対策まで」で使う
ハンズオン用のサンドボックスです。ZAPで脆弱性の疑いを見つけ、手動攻撃で情報漏洩を
再現し、最後に対策後の挙動を確認する流れを想定しています。

## 対象者

- SQLの基本文法（`SELECT` / `INSERT` / `WHERE`）を読んだことがある人
- Web開発でDBを使ったことがある人（言語・フレームワークは問わない）
- SQLインジェクションという言葉は知っているが、手を動かしたことはない人

SQL文法そのものの解説は行いません。SQLインジェクションの仕組みと対策に集中します。

## 演習の進め方（4ステップ）

各フォルダはこの4ステップで進めます。**攻撃して終わりにしない**のがポイントです。

1. 🔍 **脆弱なコードを読む**（`app.py`）→ どこが危ないか予想する
2. 💥 **攻撃してみる** → ZAPで疑いを見つけ、README記載のペイロードで突破・漏洩を確認する
3. 🔧 **自分で修正する** → `app.py` の該当箇所をプレースホルダに書き換える
4. ✅ **再攻撃して防御を確認する** → 同じペイロードが効かなくなることを確認する

答え合わせ用に `secure_app.py` を用意していますが、**まず自分で直してから**見ると
理解が深まります。各フォルダのREADMEに「理解してほしいこと」もまとめてあるので、
攻撃を試したあとに読んで振り返ってください。

## ⚠️ 注意事項（必読）

- ここに入っているアプリは**学習目的でわざと脆弱に作られています**
- **自分のPC上で `127.0.0.1` に対してのみ**動かしてください
- 許可のない他人のサーバーやネットワーク上のシステムに対してこれらの手法を試すのは
  **不正アクセス禁止法などの法律に違反する行為**です
- 各フォルダは独立したFlaskアプリです。ポート番号が被らないように分けています

## ZAPの使い方

- 対象は必ず `http://127.0.0.1:ポート番号` のローカルPoCだけに限定する
- Spiderで画面・パラメータを収集し、Active Scanで `SQL Injection` などのAlertを確認する
- Alertsの `Evidence`、HistoryのRequest/Response、画面上のエラーや応答時間を見比べる
- ZAPの検出結果は入口。最後は手動ペイロードで「本当に漏洩できるか」を確認する
- DB種別はZAPが常に正確に断定するものではない。Technology Detection、SQLエラー、コメント記法、
  時間差関数などを合わせて推測する

## 必要な環境

- Python 3.9 以上
- pip
- Docker Desktop / Docker Engine（MySQL・MongoDB PoCで使用）

## 共通セットアップ

各フォルダごとに仮想環境を分けたい場合：

```bash
cd 01_login_bypass   # 各フォルダで同様に
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python app.py
```

面倒な場合はシステムのPythonにそのまま `pip install flask` してもOKです。

## Docker Compose セットアップ（MySQL / MongoDB）

```bash
cd sandboxes/poc_sql_injection
docker compose up --build
```

## フォルダ構成

| フォルダ | 内容 | ポート |
|---|---|---|
| `01_login_bypass/` | 基本編：ログインバイパス（`' OR '1'='1'`） | 5001 |
| `02_union_based/` | 基本編：UNIONベースでの情報抽出 | 5002 |
| `03_blind_injection/` | 発展編：ブラインドSQLi（Boolean / Time-based） | 5003 |
| `04_advanced_second_order/` | 発展編：スタック型 / セカンドオーダー | 5004 |
| `05_mysql_realdb/` | 実DB編：MySQLでのUNION情報漏洩 | 5005 |
| `06_mongodb_nosql/` | NoSQL編：MongoDB NoSQL injection | 5006 |
| `07_challenge_blackbox/` | 総合演習：コードを見ずに脆弱性を探す | 5007 |

各フォルダには

- `app.py` … **脆弱な**実装（まずこれを攻撃し、その後自分で修正する対象）
- `secure_app.py` … プレースホルダで修正した**答え合わせ用**の実装
- `README.md` … シナリオの説明・ペイロード例・「理解してほしいこと」

`07_challenge_blackbox/` は総合演習用なので、修正版ではなく講師向け答え合わせをREADME内に
用意しています。受講者にはまず答え合わせ部分を見せずに進めてください。

が入っています。`app.py` を攻撃して突破できることを確認したら、**その場で `app.py` 自体を
プレースホルダに書き換えて**、同じペイロードが効かなくなることを確認してください。
自分で直せたか不安なときだけ `secure_app.py` を見て答え合わせをしましょう。

DB（`*.db`）は起動時に毎回初期化されるので、壊しても再起動すれば元に戻ります。
MySQL/MongoDB PoCのデータも講義用の架空データです。必要に応じて `docker compose down -v`
でコンテナボリュームごと初期化できます。
