# OSSへの貢献とは

---

# 「OSS貢献」と聞いて思い浮かぶもの

多くの人がこんなイメージを持っているんじゃないでしょうか：

- **Pull Request**を送る
- **コード**を書く（機能追加・バグ修正）
- **ドキュメント**を整備する

これらはもちろん立派な貢献です。

でも――**それだけじゃない**。

---

# コードの壁に阻まれていませんか？

「コードを書けないとOSSに貢献できない」

そう思って諦めている人は多いんじゃないでしょうか。

**でも、それは違います。**

OSSへの貢献の仕方はもっと多様です。

---

# 実際はもっと広い

- **バグ報告**（Issueを立てるだけでも立派な貢献）
- **ドキュメントの改善**（typo修正から構成提案まで）
- **テストへの参加**（バグを見つけるのもテスターも貴重）
- **質問に答える**（Discussionsなどで初心者を助ける）
- **翻訳**（READMEやドキュメントを日本語化）

コードを一行も書かなくても、**貢献できる方法はたくさんあります**。

---

# そして――脆弱性報告も貢献です

ここが一番伝えたいことです。

**脆弱性を発見し、報告すること。**

これも立派なOSSへの貢献です。

- 機能追加と同じくらい、いやそれ以上に**プロジェクトを守る**重要な行為
- 修正されるまでのプロセス全体が**協調的開示（Coordinated Disclosure）**
- 報告者の名前が**Security Acknowledgments**に残ることも多い

---

# ちょっとした好奇心の「ハッキング」も貢献になる

「自分の技術力を試してみたい」

そんな好奇心から、なんとなくコードを読んで穴を探す。

いわゆる**「ハッキング行為」**も、結果としてOSSのセキュリティを向上させます。

- もちろん悪用するのはダメ
- 見つけたら報告する、これが大前提
- でも**探すこと自体はOSSへの貢献の第一歩**

**たぶん、これかなり大事なポイントだと思います。**

---

# 脆弱性報告の流れ

実際の流れ：

1. **脆弱性を発見**（コードリーディング・ファジングなど）
2. **報告**（GitHubのSecurity Advisoryや専用窓口へ）
3. **開発者と非公開で議論**（修正方法・影響範囲の確認）
4. **修正される**
5. **公開・CVE発行・クレジット記載**

このプロセス全体が **Coordinated Disclosure（協調的開示）** と呼ばれ、OSSセキュリティの基本的な考え方です。

---

# 報告のハードルは想像より低い

「間違ってたらどうしよう…」

最初は誰でもそう思います。でも：

- 大抵の開発者は**感謝してくれる**
- 誤報告でも「そこは大丈夫です、でも気づいてくれてありがとう」
- むしろ**報告文化を歓迎**するプロジェクトが多い

**報告しないことの方がリスク**という認識が広がっています。

---

# OSS貢献のイメージ

```
コード寄与 ─── ガッツリ系（書ける人はやる）
バグ報告   ─── ほどほど系（気づいたらやる）
Issue     ─── ライト系（使ってて思うことを書く）
脆弱性報告 ─── 探検系（好奇心が原動力）
使うだけ   ─── 誰でもできる（最大の貢献の一つ）
スター     ─── 一瞬でできる（開発者のモチベに）
```

**どれも立派な貢献です。**

# OSSコミュニティの面白さ

---

# 開発者と知り合う

脆弱性を報告したり、Issueでやりとりしたりするうちに、自然と開発者との距離が縮まります。

- GitHub上でのやりとりが**日常に**
- リプライが来るたびにちょっと嬉しい
- 「報告ありがとう！助かりました」の一言で**やる気が続く**

---

# オンラインからオフラインへ

GitHubの画面上でしか見たことなかった名前の人が、**イベントに来ている**。

「あ、この人だ！」

- アイコンで覚えていた顔が実体化する瞬間
- オンラインの会話の**続き**ができる
- その場で**深い話**ができるのがオフラインの良さ

---

# コミュニティが連鎖する

OSSコミュニティの面白いところは、**繋がりが連鎖する**こと。

```
OSS-A で知り合った開発者A
  → 開発者Aが関わる OSS-B を紹介される
    → OSS-B で新しい開発者Bと知り合う
      → さらに OSS-C へ…
```

「このOSS、君の好きそうな分野だよ」
そう言って紹介されるOSSほど、**ドンピシャで面白い**。

---

# 気づいたら知り合いが増えている

- OSS活動をするにつれて、**知り合い・友人も増える**
- 年齢も立場も関係ない
  - 中学生、高校生、大学生、エンジニア、経営者
  - **OSSの前ではみんな対等**
- 一つのコミュニティに属さなくていい
  - **興味の赴くままに**渡り歩けるのがOSSの世界

---

# OSSコミュニティは「人との繋がりの宝庫」

ソフトウェアを作る場所であると同時に――

**人との繋がりを作る場所。**

これが僕がOSSコミュニティにハマった一番の理由かもしれません。

# なぜ登壇するのか

---

# 動機は単純：オフラインで会いたいから

OSS活動をしている人たちの中には、当然**カンファレンスに出ている人**もいます。

- オンラインだけの関係じゃ物足りない
- **実際に会って話したい**
- 画面の向こうの人と**直接交流したい**

そのための一番の方法が――**登壇**でした。

---

# 登壇すると覚えてもらえる

- 「あ、あの登壇してた人だ！」
- **顔と名前が一致する**
- GitHubのアイコンだけだった自分に**実体が宿る**

聴衆の**記憶に残る**。これが次の出会いを生みます。

---

# 登壇から広がる循環

```
登壇
  ↓ 認知される
  ↓ 声をかけられる
  ↓ 新しい繋がり
  ↓ 新しいOSSとの出会い
```

登壇はゴールではなく、**次の出会いのきっかけ**です。

---

# 正直に言うと――

**驚かせたい**という気持ちもあります。

- 「高校生がこんなことやってるんだ」
- 「面白いことやってるな」

そう思ってもらえたら、話しかけてもらいやすくなる。
結果として、**新しい人脈ができる**。

**これは戦略です。**（笑）

---

# でも失敗は怖い

当然、失敗は怖いです。

- 噛む
- 緊張で頭が真っ白になる
- 時間を超過する
- 滑る

**失敗を恐れることは大事です。**
恐れずに突っ走るのはただのバカですから。

---

# でも――所詮、この程度の話

ここからは少し哲学っぽくなります。

**所詮、僕らが図り切れるはずもない世界。**
そんな世界にいる僕らという矮小な「個人」。

**所詮、いつ生まれようがいつ死のうが、それもまた人生。**

このスケールで考えたら――
**LTで噛むとか、時間超過するとか、そんなの本当に些細なこと。**

---

# 死んだ後の話をすると―

- 死んだ後、意識があるのなら → それで一つの仮説が正しくなる。面白い。
- 死んだ後、意識がないのなら → 永遠に苦しみを味わわずに済む。それもよし。
- もし永遠に一人で過ごすことになったとしても → その結末がとても気になる。面白そう。

**どっちに転んだって面白いんですよね。**

――という思考にたどり着くと、**今この瞬間の恥ずかしさなんて、どうでもよくなる。**

---

# 失敗は飲みのネタになる

実際に僕は結構イベント終わりの懇親会で**失敗談を話します**。

（もちろん僕は未成年なので飲酒はしてないですけどね？！）

- 「この前のLTでこんなやらかしました」
- 笑い話になる
- その失敗談自体が**また別のLTのネタになる**

**成功するも、失敗するも。登壇経験というのは、特にしかなりません。**

---

# それでも終わる

どんなに失敗しても――

**発表は必ず終わります。**

- 噛んでも終わる
- 緊張しても終わる
- 時間超過しても終わる

終わった後は、**次はもっと上手くできる**と思うだけ。

---

# 登壇は経験値

一度登壇すると、二度目は少しだけ怖くなくなります。

- 1回目 → めちゃくちゃ緊張
- 2回目 → まだ緊張するけど、何とかなることを知っている
- 3回目 → 「まあ何とかなるだろ」

**登壇はレベル制のゲームみたいなもの。**
何度もやるほど強くなります。
